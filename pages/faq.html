Adhearsion is for Ruby on Rails?

This is one of the biggest misconceptions about Adhearsion. It should be strongly stressed that Adhearsion is a Ruby framework, not a Ruby on Rails framework. Ruby on Rails, a web development framework, is probably the most popular Ruby software, but it has little to do with the Ruby language per se besides simply using it. The Ruby language is a general purpose programming language with which both Ruby on Rails and Adhearsion are written.

Adhearsion does help facilitate integration with Rails applications and it shares a few libraries with Rails, such as ActiveSupport and ActiveRecord.

Why Ruby?

For a couple reasons:

  • Ruby strongly promotes code-reuse through first-class Object Oriented Programming.
  • Ruby code emphasizes being clean and concise
  • Ruby has many interpreters, drastically extending the code-reuse potential. The main alternative interpreter is JRuby, a stable interpreter capable of running Adhearsion atop the JVM, but other interpreters are in the works for .NET, Objective-C and Smalltalk.
  • Ruby has a fantastic package manager named RubyGems which makes code reuse even easier and simpler.
  • The Ruby community is full of very intelligent, pragmatic individuals with a culture like none other. Things such as Agile software development
  • Adhearsion wants to attract people who get excited about learning new things.

How mature is the project?

Adhearsion is production-worthy but, like most software, don't consider it to be totally bug-free. Adhearsion was first released on Christmas of 2006 publicly, though its inception dates many months prior to that.

Does Adhearsion run on Windows?

Not officially.

Why doesn't Adhearsion support Ruby 1.9?

Adhearsion works with Ruby 1.8.x versions of Adhearsion (starting at 1.8.6) and JRuby 1.1 or later. JRuby has the distinct advantage of being able to run Adhearsion on the Java Virtual Machine, allowing you to use virtually any Java library in existence right in your Adhearsion application. The normal Ruby interpreter has fewer dependencies and is what's generally used in Adhearsion applications at the time being.

Do I have to recompile Asterisk to use Adhearsion?

Nope!

Can I try Adhearsion safely on my production Asterisk machine?

Yep! You can have a special SIP account route exclusively to Adhearsion very easily, leaving the rest of the logic perfectly intact. You can even have a single extension route to Adhearsion if you'd like.

Can't guarantee that Asterisk won't die (likely through race conditions causing a segfault) or suffer memory leaks from Adhearsion using its API. For example, the Asterisk Manager Interface feature of Asterisk (which Adhearsion optionally uses) used to cause crashes. Nothing in Adhearsion is particularly abusive, but Asterisk has been known to arbitrarily fall over. For a production phone system, you should always consider using a process watchdog such as monit.
